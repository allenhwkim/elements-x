Why We Should Avoid Modal Dialog

I have written an article about how we properly use modal dialog, https://medium.com/allenhwkim/modal-or-not-modal-3d65de64d82c.

This article, I talk about only negative sides of modal dialogs.

When you make it as a dialog, the contents is not navigable by URL because the state of dialog is not representable by URL.

I do not find any reason we should intentionally make the state not navigable. For example, I want to save the payment page url, so that I can easily pay when I need it. But, but, it’s a dialog. I have to click it to see the dialog.

Let’s imagine this conversion.

A: Hi, I just sent you an url, so that you can pay online. B: Thanks, I clicked, but I don’t see it. A: You should click a link to see payment popup. B: Which link? there are so many links. A: Link on the top right side of your monitor! It says “Make Payment”. B: Are you on desktop? It looks different on iPad. A: Uhhokaaay….

From Facebook changes, I think NOT using modal dialog is a trend now.

From the article, The current state of modal dialog accessibility

Modal dialogs continue to be troublesome UI components all across the web. Putting aside the fact they are often misused and thrust on users in a manner that interrupts their current task (asking me to sign up for your newsletter, while I’m in the middle of reading an article, is not cool), even appropriately used modals often lack considerations for accessibility.

For the sake of accessibility, it’s a programming nightmare.

Based on W3C standard and webAIM, a modal must have the following accessibility requirement. Modal dialogs contain their tab sequence. That is, Tab and Shift + Tab do not move focus outside the dialog. However, unlike most non-modal dialogs, modal dialogs do not provide means for moving keyboard focus outside the dialog window without closing the dialog.

  • Modal dialogs should be navigable with ‘Tab’ and ‘Shift-Tab’ key.
  • Modal dialogs should be closable with ‘Esc’ key and close button.
  • Modal dialogs maintain focus until dismissed.
  • Modal dialogs should maintain keyboard focus.
  • Non-modal dialogs should close automatically when they lose focus.
  • When a dialog closes, focus should usually return to the element that opened the dialog.
  • When dialog is open, the underlying page contents should not be scrollable.
  • Modal dialogs should have role=“dialog”, aria-modal=“true” and aria-label to dialog title

    • Problems

  • The underlying page must NOT be scrollable and must NOT be focused with ‘Tab’ while a modal dialog is open. This script will make your code complicated if you don’t use a well-tested framework.
  • What if a component in a modal dialog should accept escape key? It should close the dialog, or should it behave as a a normal escape key? For example we can have a dropdown menu in a dialog, and we should use ‘Esc’ key to close the dropdown menu. On this case should we ignore ‘Esc’ key of closing dialog, or closing dropdown menu? You will need another keyboard controlling Javascript, which makes your code more complicated.

    • How do you display full page dialog contents in tablet or mobile devices properly?

    Mobile first! However, Modals and mobile devices usually don’t play well together. Viewing the content is difficult because modals either are too large, taking up too much screen space or too small. The following is example of modal dialog makes users confused when it comes to tablet or mobile devices.

  • How do we scroll properly within dialog without the underlying page scrolling?
  • Should dialog cover header and footer too? If not, dialog should be between header and footer?
  • Why do we bother users by these things by making it as a dialog?

    • Performance Degrade

    It’s not easy to lazy load modal dialog codes including js, css, html, and library used by modal without extra programming. No matter user open a modal or not, the code for modal dialogs must be always ready to take an action from user, so that it can immediately open the dialog. Let’s imagine that you have 3 to 4 modal dialogs to open on your dashboard page. Your dashboard page will be loaded much more slowly with modal dialogs although there is very little possibility user opens all modal dialogs.

    Conclusion

    Modal windows are a useful and effective interface widget to use when designers use them right. Use modal dialog with this in mind

  • The modal dialog is not navigable in URL nature.
  • The modal window should have a Cancel/Close button.
  • ‘Esc’ key should close the window
  • Close when clicking outside the modal window
  • Modal dialogs maintain focus until dismissed.
  • Don’t put too much contents into modal dialog.
